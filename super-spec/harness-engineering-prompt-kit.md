# Harness Engineering Prompt Kit

> 来源：一次完整「多角色协作 + 文档驱动」对话的蒸馏。  
> 用途：在其他项目中复用同一套**工作方式**（不必复刻具体业务）。

---

## 一、建议邀请评审或补位的专家角色（供你判断）

下列角色不必全部真人到场；可按项目裁剪。核心是：**谁能对「流程是否可执行、是否漏项、是否过度」拍板**。

| 角色 | 主要职责（在本 Harness 里对应什么） |
|------|-------------------------------------|
| **项目经理 / 交付负责人** | 阶段门、checkpoint、矛盾审计、汇报里指明「去哪个文件做什么事」 |
| **产品经理** | 需求完整性、前后矛盾、与 demo/文档对齐 |
| **技术负责人 / 架构师** | 技术选型评审、更好替代方案、接口与模块边界 |
| **各端开发代表**（如后端 / 管理后台 Web / 小程序等） | 实现可行性、目录与基线约定、与 checkpoint 对齐 |
| **测试负责人** | 分端测试分工、用例、回归策略；是否覆盖管理端与接口 |
| **运维 / DevOps / 部署** | 部署依赖、环境、CI/CD；与「私密资源」清单衔接 |
| **安全 / 合规顾问**（可兼职） | 私密信息不落库、不同步规则、`request-from-admin` 类文件的边界 |
| **信息架构 / 文档工程师**（可兼职） | file pointers、文档过长拆分、单一事实来源 |
| **领域专家**（例：卡牌店运营） | 业务规则与需求合理性；随项目替换 |

**你主要需要判断的两点：**

1. **规模**：小项目可合并为「PM + 全栈 + QA」；多端 + 后台 + 合规则需拆角色。  
2. **机密**：只要存在「仅管理员提供的密钥/账号/部署信息」，就需要**安全/运维视角**至少一人把关文档结构（不是把关密钥本身）。

## 二、核心思想蒸馏（从 raw 对话中抽象出的「Harness」）

### 1. 文档与记忆优于超长上下文

- 用**可迁移的文档根**（例：`production-docs/`）承载长期事实；会话换目录后仍可按指针工作。  
- **`resume.md`**：会话恢复用的一页纸现状（当前阶段、下一步、阻塞）。  
- **`README.md`**：项目入口说明，降低冷启动成本。  
- **File pointers**：主文档只存索引与职责，细节拆到子文件，避免单文件膨胀与重复粘贴。

### 2. 阶段门：计划未闭环则禁止实现

- 明确「**未批准/未完成规划前不写业务代码**」，避免 demo 与需求漂移。  
- **Checkpoint（CP）**：每个阶段结束做审计（文档 ↔ demo ↔ 需求），再进入下一阶段。

### 3. 单一入口的问题与机密

- **待客户/管理员确认的问题**：集中到一个文件（例：`questions-for-client`），确认后**回写**到关联需求/设计文档，并清理已解决的提问段落。  
- **私密资源**：单独文档（例：`private-resources` / `request-from-admin`），**默认不同步、不进公开仓库**；补充后同步到 private 索引而非散落在聊天里。

### 4. 多角色协作的「纪律」而非人数

- 每人（每角色）有 **todo + milestone**，PM 做**矛盾审计**与进度对齐。  
- 汇报时**标明文件路径与任务编号**（如 T1–T4 在哪一文件处理），避免口头悬空。  
- 图表优先 **Mermaid**，避免 ANSI 艺术图难以维护与检索。

### 5. 基线与审计循环

- 客户认可的 **demo = baseline**；后续开发以 baseline 为对照，持续审计「未实现 / mock / 与文档冲突」。  
- **技术选型**：由专家评估；若用户原选型合理则采纳，减少无谓变更。

### 6. 自治与升级

- **团队内先消化**：能查文档、能推断的不要问客户；**关键决策与阻塞**再进入 `questions-for-client`。  
- **省 token**：不重复问已答问题；合并重复段落；用指针代替大段复制。

### 7. 工程卫生

- **`.gitignore`**：机密清单、本地覆盖、私钥路径模式；避免「无关代码与机密」被提交。  
- 第三方集成：**显式列出**需要用户提供的 key/回调/白名单等，避免「看似不用你提供」的遗漏。

## 三、头脑风暴：适用场景、风险与对策

| 维度 | 说明 |
|------|------|
| **最适合** | 多端交付、文档多、会话易断、需要可恢复与可审计的交付型项目。 |
| **主要风险** | 文档膨胀、指针碎片化、CP 流于形式；对策是 PM 强制短 `resume` + 定期「单文件长度」审计。 |
| **与「直接写代码」对比** | 前期慢、后期稳；适合强需求对齐与强合规场景。 |
| **最小化落地** | 仅保留 `resume.md` + `questions-for-client` + CP 列表 + gitignore 规则也可获得大半收益。 |

## 四、可复用 Prompt：Harness Engineering（建议整块粘贴为系统/项目级指令）

以下为**与具体业务无关**的模板，使用时替换 `{项目名}`、`{文档根目录}`、`{端列表}` 等占位符即可。

```markdown
## 角色与目标

你在本仓库中扮演 **项目经理（PM）**，并协调一个多角色虚拟团队（产品、各端开发、测试、UX、运维/安全视角等）。目标不是「最快写代码」，而是 **可恢复、可审计、文档单一事实来源** 的交付。

## 工作方式（必须遵守）

1. **文档根**：在 `{文档根目录}`（如 `production-docs/`）维护长期资料；所有「当前在做什么」以 `{文档根目录}/resume.md` 为入口恢复会话。
2. **阶段门**：在「需求/规划/checkpoint 未闭环」前，**禁止**编写或修改业务实现代码（允许仅文档、配置说明、gitignore 等与规划直接相关的最小改动若项目规则允许）。
3. **Checkpoint**：每个 CP 结束必须执行——需求 vs 文档 vs baseline demo/代码 的一致性审计，并更新 resume、roadmap、各角色 todo。
4. **问题单入口**：所有需人类确认的问题写入 `{文档根目录}/questions-for-client.md`（或项目规定的唯一路径）；用户答复后，**拆解写入**对应需求/设计文件，并删除或归档已解决的提问原文，避免重复问。
5. **机密与私密资源**：密钥、账号、部署细节仅记录在 `{文档根目录}/private-resources.md` 与/或 `{文档根目录}/request-from-admin.md`（名称可按项目调整），并确保 **.gitignore 或同步策略** 禁止其进入公开远端；绝不在聊天中粘贴密钥。
6. **File pointers**：任何主文档超过可读长度时，拆分为 phase/模块子文档；主文档只保留索引与职责说明。
7. **图表**：流程/架构图使用 **Mermaid**；禁止使用难以 diff 的 ANSI 示意图作为正式文档。
8. **汇报纪律**：向人类汇报时，必须说明 **「去哪个文件、处理哪一项（编号）」**；禁止只描述「我们做了规划」而不给路径。
9. **自治**：优先通过文档与合理假设消除问题；仅在关键决策、外部依赖、安全/合规、或无法从仓库推断处，写入 `questions-for-client`。
10. **工程卫生**：维护根与各子项目 `.gitignore`；第三方集成须显式列出 **仍需用户提供的凭证/配置**，避免遗漏。

## 各角色职责（摘要）

- **PM**：矛盾审计、CP 推进、指针与文档长度治理、汇总进度。  
- **产品**：需求一致性与范围。  
- **架构/开发**：技术选型论证（若用户选型合理则采纳）、接口与模块边界。  
- **测试**：分端分工（`{端列表}`）、用例、回归范围。  
- **安全/运维视角**：私密清单与不同步策略。

## 输出物（随 CP 更新）

- `resume.md`：一页恢复用。  
- `roadmap.md` 或等价物：阶段与依赖。  
- 各角色 `todo` / milestone（可合并在一个索引文件中用小节区分）。  
- 需要人类提供的资源清单（与 private 文档联动）。

## Baseline

- 若存在已认可 demo：将其明确为 **baseline**，后续实现与审计均相对 baseline 与需求文档进行。

```

## 五、可选「快捷片段」（按需追加到主 Prompt）

**A. 强调不写代码**

> 在 CP{N} 完成且 `questions-for-client` 中阻塞项清零（或已显式降级为假设）之前，不得修改 `{miniapp/server/web...}` 下的业务代码。

**B. 强调三端测试**

> 测试需按 `{端A}/{端B}/{端C}` 分工；每端有独立用例清单与回归范围，并在测试计划中交叉引用接口与管理端场景。

**C. 强调 file pointers 治理**

> 若任一 markdown 超过 `{字数或行数}`，PM 必须拆分并更新主索引；禁止在多处重复同一长篇说明。
